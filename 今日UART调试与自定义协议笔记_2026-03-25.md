## 1. 查阅与梳理：UART 相关代码位置

### 1.1 UART 底层例程（Ring Buffer / DMA / 收发）
- 文件：`D:\YiChip_workspace\Stack\program\peripherals.prog`
- 重点函数（UARTA）：
  - `uarta_init_dma_mem` / `uart_init_dma_mem`：初始化 RX/TX Ring Buffer 地址与指针，并打开 UART 时钟
  - `uarta_prepare_tx` / `uarta_prepare_rx`：取出 TX/RX 的 start/end/读写指针供软件使用
  - `uarta_send`：提交 TX 写指针（写回 `core_uart_twptr` 触发发送）

  - `uart_copy_tx_bytes(_fast)` / `uart_copy_rx_bytes(_fast)`：在 ring 与普通内存之间搬运数据
  - `wait_uarttx`：等待 TX buffer 发送完

### 1.2 两个“上层”文件的定位差异

#### `app_module.prog`
- 文件：`D:\YiChip_workspace\Stack\program\app_module.prog`
- 定位：模块业务逻辑（BLE 事件 ↔ UART HCI-like 命令/事件桥接），包含 LPM/唤醒/状态 GPIO 等。
- `module_lpm_uart_init` 做的事：
  - 关闭 UART → 配 DMA/Ring → 配波特率 → 选时钟 → 配 GPIO 复用 → 使能 UART（可选流控）

#### `hci_main.prog`
- 文件：`D:\YiChip_workspace\Stack\program\hci_main.prog`
- 定位：偏“纯 HCI 主循环/调试模式”，初始化 UART 后进入轮询收包解析。
- `hci_init_uart_config` 做的事：
  - 配 ring buffer → 选时钟 → 用 115200 计算 baud 配置 → 按 `mem_hci_uart_*_gpio` 配 GPIO → 使能 UART

### 1.3 HCI 概念说明（为什么不能随便发字符串）
- HCI 是 Host↔Controller 的协议接口。
- 工程中存在 HCI/HCI-like 格式解析：例如要求首字节 `HciType==0x01`，不符合会丢弃/异常处理。
- 结论：串口是否“能看到回显”要看固件是否做透传；若跑协议栈，必须按协议格式发包。

## 2. PinMux 与“全映射”理解

### 2.1 全映射的道理
- 所谓“所有引脚都可配成串口脚”，本质是芯片内部有引脚复用矩阵（PinMux / Switch Matrix）。
- 软件通过写 GPIO 配置寄存器（`core_gpio_conf` / `core_gpio_conf_hi`）把 UART 信号路由到指定 GPIO。

### 2.2 工程内对 UART 默认脚位的定义
- 文件：`D:\YiChip_workspace\Stack\format\hci.format`
- 默认：
  - `HCI_UART_TX_GPIO_NUM = 0x07`
  - `HCI_UART_RX_GPIO_NUM = 0x06`
  - `HCI_UART_RTS_GPIO_NUM = 0x09`
  - `HCI_UART_CTS_GPIO_NUM = 0x0a`

### 2.3 关键坑：GPIO16 属于“扩展段”
- 工程里 GPIO 配置分两段：
  - `core_gpio_conf`：基础段
  - `core_gpio_conf_hi`：扩展段（GPIO>=16 常见走这里，并做 `-16` 偏移）
- 直接写 `jam xxx,core_gpio_conf+16` **可能写错寄存器表**，导致 GPIO16 实际没有被切到 UART_TXD。

## 3. Ring Buffer 是什么、怎么初始化、怎么收发

### 3.1 Ring Buffer 定义
- TX/RX 各一段连续 RAM，硬件 DMA 自动维护读写指针，指针到尾部会回环。
- 软件要做的核心动作：
  - TX：写入 ring → 提交写指针（写回 `core_uart_twptr`）
  - RX：读出 ring → 提交读指针（写回 `core_uart_rrptr`）

### 3.2 初始化要求（4 个地址打包为 8 字节）
`uarta_init_dma_mem` 需要按顺序提供：
1) RX start（16bit）
2) RX end（16bit）
3) TX start（16bit）
4) TX end（16bit）

工程示例参考：`D:\YiChip_workspace\Stack\program\hci_main.prog:hci_init_uart_config`

## 5. 今日遇到的关键问题与定位/修复

### 5.1 问题现象
按键触发发送后，串口调试助手无接收数据。

### 5.2 排查结论
主要原因是 GPIO16 的配置方式：
- 之前用 `jam gpcfg_uart_txd,core_gpio_conf+16` 配 TXD
- 但 GPIO16 实际属于扩展段，应走 `core_gpio_conf_hi` 表
- 导致 UART_TXD 没有真正输出到 GPIO16，引脚层面就“没信号”

### 5.3 修复方法
改用 `gpio_config_function_int` 配 GPIO15/16：
- 函数内部会根据 GPIO 号自动选择 `core_gpio_conf` 或 `core_gpio_conf_hi`，并正确处理 `-16` 偏移。
- 修改后发送立即恢复正常。

## 6. 自定义协议：2 字节消息包 + 覆盖式 16 字节堆栈

### 6.1 协议定义
每个消息包固定 2 字节：
- `cmd0 == 0x00`：将 `cmd1` push 进堆栈（满了覆盖最老数据）
- `cmd0 == 0x01`：pop 1 字节并通过串口发送；若空栈发送 `0xEE`

### 6.2 覆盖式堆栈实现思路
- 使用环形索引维持“LIFO + 满了覆盖”：
  - `top_idx`：下一次 push 写入的位置（0~15）
  - `count`：当前元素数量（0~16）
- push：
  - 写 `stack[top_idx]=value`
  - `top_idx=(top_idx+1)&0x0F`
  - `count=min(count+1,16)`
- pop：
  - 若 `count==0`：空
  - `top_idx=(top_idx-1)&0x0F`
  - 读 `stack[top_idx]`
  - `count--`

### 6.3 已落地的接收模块
- 文件：`D:\YiChip_workspace\Stack\program\patch.prog`
- 新增函数：
  - `my_uart_receive_cmd2`：按 `core_uart_rxitems>=2` 读取 2 字节并分发处理（循环把当前 RX 中已有包处理完）
  - `my_cmd_stack_push_overwrite`
  - `my_cmd_stack_pop_and_send`
  - `my_uart_send_tx_byte`
- 主循环已改为调用：`my_uart_receive_cmd2`
- 初始化已加入：`mem_cmd_stack_top_idx/count` 清零（在 `p_my_feature_init` 中）

### 6.4 对应的 format 变量

  - `mem_my_uart_tx_byte`
  - `mem_my_uart_cmd0` / `mem_my_uart_cmd1`
  - `mem_cmd_stack_space`（16 字节）
  - `mem_cmd_stack_top_idx` / `mem_cmd_stack_count`

## 7. 串口调试助手验证方法（推荐）

### 7.1 串口参数
- 波特率：115200
- 数据位：8
- 校验：None
- 停止位：1
- 流控：None（除非你显式启用了 RTS/CTS 并接线）

### 7.2 协议验证（必须十六进制发送）
按 2 字节一包发送：
- push 0x33：`00 33`
- push 0x44：`00 44`
- pop：`01 00`

期望接收（pop 的回包）：
- 依次收到：`44`、`33`（后进先出）
- 空栈 pop：收到 `EE`
